Defining Project Scope

To ensure Scanny Assist works properly, you must accurately define the project scope. Failure to do so will mean Scanny Assist will not collect data to assist you during the assessment. Once the scope is defined, you will see the tables within the main tabs of Scanny Assist start to populate:

Scanny Assist only records endpoints with parameters. These parameters can be any of the following:

  • XML

  • JSON

  • Form Urlencoded

  • Form Multipart

  • Query

  • Numerical/UUID Path

When we go to the Manual, Scan and Parameters tabs we can see a more in depth breakdown of the parameters we have.

Opening Burp

When you first open Scanny Assist, you will be given a few prompts for some data. The first of these is for the NetSPI pentest header.

Enter your username (e.g. for Tyler Sullivan, it would be tsullivan). There are more details about this header under NetSPI Header.

The next of these is to modify the project options.

This applies NetSPI’s collaborator instance to the current Burp project. If for client reasons you need to use the basic instance or if the assessment is internal, you can opt to keep the basic project options.

The final option is to update the Burp Store extensions.

This installs the base set of extensions required for WaPen at NetSPI. If they’re already installed, nothing will change in your environment. If they need to be installed, Scanny Assist will download them from the store and then reload all extensions; you may see extensions disappear and reappear.

Tabs

Overview

The overview tab (as the name suggests) is designed to provide an overview of the current state of testing, providing statistics as well as a summary of all the endpoints seen so far. Below is what your Overview tab might look like:

Statistics

The Automated Testing Coverage piechart and Test Statistics can provide yourself with a quick overview of the state of testing so far.

Completion is determined by the number of times Scanny Assist observes different payloads for a parameter, through manual means or automated means (details of which are in the next section).

The thresholds for completion are static, 12 for manual and 50 for automated (scan). As these are static, they may not always reflect your testing requirements, but can be seen as a guide to see how you are getting on for a certain endpoint or parameter.

Crawler Enumeration

At present, there is really only one option for Scanny Assist which is the ability toggle endpoints that were enumerated via automated means (such as Burp Scanner or Crawler). By default it is off, meaning you’ll only see endpoints enumerated through manual navigation of the application. The reason behind this is Crawler and Scanner can often create arbitrary endpoints that the server responds with 200 OK and it can quickly fill the Endpoint tables with junk. Any endpoint that is enumerated via automated and verified manually (e.g. by browsing to it) will be visible in the default view, without switching on the crawler toggle.

Manual & Scan

These tabs are almost identical, with the exception one looks at manual coverage and the other scan. We will look at the Manual tab, but everything here applies to Scan equally. He is an overview of what the tab may look like:

Testing Details

This section provides a list of every endpoint with parameters within your defined project scope, along with some data about the number of times the endpoint and its parameters have been tested.

The table has a list of the missed parameters, i.e. the ones that have not yet reached full coverage. You can also view any parameters marked out of scope which will get into shortly. You can double click and right click these endpoints the same as the Overview tab.

Right clicking and pressing “Mark All Parameters Out of Scope” will move all the parameters for that endpoint to the “Out of Scope” column, effectively marking the endpoint as complete:

Parameters

Clicking on the endpoint in the top table will populate the parameters table below. This includes a breakdown of all the parameters.

Firstly, the parameter identifier is used to describe each parameter uniquely. For example, if you have multiple instances of parameter “abc” in JSON, both parameter names will be “abc”, but their identifier will include their nested path and optionally a unique identifier at the end:

Often these will be the same value for things such as form data. Alternatively, there may be no parameter name where the parameter is in the URL. In these cases, the identifier can be used to determine which parameter in the URL is being identified. In the example below, the URL replaced “7” with “{ID}”. and the parameters identifier states it is the third parameter in the URL:

The originating request column shows the first request that revealed this endpoint. This is particularly useful if the application uses the same endpoint to send different data, like in GraphQL. To view this originating request, you can double click the parameter:

Note: the response for each parameter is not stored for Burp file size reasons.

There are additional options when you right click a parameter:

You can granularly mark single parameters in or out of scope, send the originating request to repeater or intruder, or send the parameter to the Parameters tab, which we will cover now.

Parameters

Without sending a parameter to the tab, it will look similar to the following:

Global Parameter Scope

in the previous tabs, you could set parameters to be out of scope for single endpoints. In the Global Parameter Scope, you can mark parameters out of scope for all endpoints. This is particularly useful for things such as CSRF tokens or __VIEWSTATE parameters. Right clicking on the global parameters has the option to put the parameters back in scope.

Parameter Details

When right clicking parameters in the Manual and Scan tabs, selecting “Send to Parameters” will populate that parameter into the Parameter Details:

Here you can see standard details for the parameter and also each of the values that has been used for that parameter. This can allow you to see what type of testing you’ve performed so far on the parameter and what still needs to be done.

Note: There are plans to make this even more useful by classifying payloads into XSS, SQLi, etc.

Parameter Search

Parameter Search lets you view and search every payload for every request seen so far. This can be useful in seeing where similar data may be submitted in the application, or if one parameter has a vulnerability, you may find the same parameter names with the same vulnerability

You can use right click and double click the same as the other parameter tables here.

NetSPI Header

The header will look like this:


X-Netspi: NetSPI Penetration Test - tsullivan - activepentest@netspi.com

This makes it easier for our clients to see which requests have come from NetSPI as part of our Web, API or Mobile tests. This is something clients have asked us for.

If you need to modify the header in any way, you can do so in the NetSPI Header tab:

As stated, it is imperative client data is not added into the headers.

By default, the header is applied. If for any reason it is not required for a test, simply uncheck the box.

One note about the email address that is appended - this is an email box that clients can see in their logs and use to contact NetSPI if there is an issue or concern. Messages received will be read and responded to which is why it is different than the securitytesting@ email addresses.
